# 60-80-100 模型：Builder + AI + 专业团队的协作阶梯

> “先把可用的 60 分交付出来，再用 80 分稳态上线，用 100 分精修传播与扩展。”

## 背景

- 目标：用最低成本交付“足够好”的成果并可持续提升，形成 60→80→100 的可控阶梯。[Source: docs/PRD.md#Executive-Summary]
- 适用：内容与产品并行（文档/PPT/代码），强调快速验证、逐步打磨与复用。

## 概念

- **角色分工**  
  - Builder：端到端 owner，驱动 AI 生成/检查/重构。  
  - AI：提供草稿、对比、一致性检查与重写建议。  
  - 专业团队：关键节点审阅，守护事实、风格与风险。
- **阶段定义**  
  - 60 分：可用、可验证的 MVP/初稿，验证方向与框架。  
  - 80 分：补齐引用、图示占位、结构一致性与基础质量。  
  - 100 分：精修语言/数据/案例，强化传播性与长期复用。

## 解释

1) 60 → 80：如何升级  
   - 用 AI 做查漏补缺、风格/一致性检查。  
   - 补齐引用与图示占位，确保结构完整（背景→概念→解释→意义）。  
   - 让专业人员做“最小必要审阅”，确认逻辑与事实。

2) 80 → 100：如何打磨  
   - 深入校对与事实核验，补充市场数据/案例引用。  
   - 优化叙事顺序、增加示例/对比，增强说服力与传播性。  
   - 将可复用片段抽离成模板/引用库，降低后续成本。

3) 质量控制与测试  
   - 每个阶段都有可交付 DoD（结构完整、引用占位、图示占位、术语一致）。  
   - 内容型交付用人工/AI 校验；代码型交付用自动化测试 + 代码审查。

## 意义

- 理解：建立“渐进式”完成定义，避免完美主义导致停滞。  
- 实践：给团队清晰的节奏与验收项，减少返工；强调复用与可迁移。  
- 传播：60/80/100 形成可复述的比喻，便于在 PPT/培训中讲解。

## 实践指南

- 阶段入口/退出标准  
  - 60：骨架齐全（背景/概念/解释/意义）、关键引用占位、图示占位；AI 缺项检查通过。  
  - 80：引用与图示清单补全，占位替换为真实数据/说明；语言/风格一致；审阅一次通过。  
  - 100：事实核验、案例/数据补充、传播友好（摘要、要点、图表）；可直接外发或演讲。
- 迭代节奏  
  - 先出 60（快），再出 80（稳），最后打磨 100（精）；每一步都有 DoD，避免“同时追求完美”。  
  - 失败回滚成本低：保持 60 版本可用，80/100 迭代不影响基本交付。
- 指标与工具  
  - Time-to-60/80/100，审阅通过率，返工率，复用率（片段/图示被引用次数）。  
  - AI 用途：缺项检查、风格统一、引用格式、重写/压缩/扩写。

## 检查清单

- 是否为每个阶段定义了 DoD（结构、引用、图示、风格、审阅）？  
- 是否记录 Time-to-60/80/100，跟踪返工与通过率？  
- 是否把可复用段落/图示抽到公共库，支持下一次 60/80 的快速生成？

## 场景与风险对策

- 场景：内容发布  
  - 60：生成骨架 + 引用/图示占位；AI 缺项检查。  
  - 80：补全引用与图示说明，审阅确认；风格统一。  
  - 100：增加案例/数据，压缩/扩写两版（文档/PPT），准备对外发布。  
  - 指标：Time-to-60/80/100，审阅通过率，复用片段数。
- 场景：产品功能说明/演示稿  
  - 60：最小演示脚本、关键截图占位。  
  - 80：添加风险与已知限制，确保一致性与可演示性。  
  - 100：打磨故事线与视觉，准备对外讲稿。
- 风险与对策  
  - 风险：停在 60 或 80，迟迟不完结 → 对策：设定时限与清晰 DoD；保持 60 可用，80/100 渐进。  
  - 风险：复用片段未沉淀 → 对策：每次迭代后将可复用段落/图示入库，记录引用路径。  
  - 风险：审阅缺口 → 对策：为 80/100 设置“守门人”与检查清单。
